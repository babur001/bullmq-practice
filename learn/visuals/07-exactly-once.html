One job. How many charges?

Worker A claims a job under a lock with a TTL, charges a card, then needs to ack "done" to Redis. If A stalls in the gap between charging and acking, the lock expires, Redis hands the same job to Worker B, and B charges again — unless a consumer-side idempotency guard (atomic SET NX) stops it. Toggle the two switches and watch the ledger.

Default (stall on, guard off) → 2 charges from one job: the canonical double-charge. Turn the guard on → 1 charge: B's atomic claim finds the key A already set and skips. Turn stall off → A finishes before the lock expires, B never gets the job, 1 charge either way. The lock is a lease, not a mutex — only the guard makes processing effectively-once.
